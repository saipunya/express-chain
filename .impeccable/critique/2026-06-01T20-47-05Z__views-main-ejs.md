## Overall Impression

The page now reads as a credible government-style operational dashboard: cleaner, modern, and useful. The best part is that the institution summary is actionable and domain-specific. The weakest part is that the screen tries to be a landing page, login page, status dashboard, and public data explorer all at once.

## What's Working

- The visual direction matches the CoopChain design intent: green/teal institutional palette, Sarabun typography, calm surfaces, and practical data sections.
- Institution summary cards are meaningful and use proper button semantics for modal disclosure.
- Desktop composition is stable: sidebar, hero, login, summary, and deadline sections have clear zones and no page-level horizontal overflow.

## Priority Issues

P1: Login task competes with too much dashboard content.  
Why: On mobile, login starts around 956px after sidebar and hero. For a user whose goal is to sign in, the page asks them to parse public dashboard context first.  
Fix: Make login the primary first-screen action on mobile, or add a sticky/early login shortcut and reduce hero height.  
Suggested command: inspect `views/main.ejs:719-781` and responsive rules around `views/main.ejs:587-644`.

P1: Some elements imply functionality or live accuracy that is not obvious.  
Why: Search is visible in the desktop snapshot but has no clear handler. Hero progress bars use fixed widths `42%` and `30%`, which can diverge from real counts.  
Fix: Implement search/filter behavior or remove the search field. Bind progress widths to real totals or replace with simple count chips.  
Suggested command: update `views/main.ejs:707-717` and `views/main.ejs:727-739`.

P2: Browser overlay flags contrast and visual-weight issues.  
Why: Several green/teal text-on-background combinations landed below 4.5:1, and many 1px border plus wide-shadow surfaces make the UI feel more like a template than a restrained government system.  
Fix: Darken small text colors, reduce shadow blur/count, and reserve strong shadows for modal/login only.  
Suggested command: tune tokens and shadows in `views/main.ejs:35-90`.

P2: Form validation and recovery state are under-specified.  
Why: `novalidate` disables native validation, but the visible template does not show inline invalid states, error copy, or recovery links.  
Fix: Add Bootstrap invalid feedback and server error rendering near the login form.  
Suggested command: improve `views/main.ejs:757-780`.

P3: Semantic hierarchy has a modal heading skip.  
Why: Overlay flagged `h2` followed by `h5` in the modal title. It is not catastrophic, but it is easy to clean up.  
Fix: Use a heading level that follows the page structure, or style a semantic `h3` as a modal title.  
Suggested command: adjust `views/main.ejs:976`.

## Persona Red Flags

- First-time cooperative user: may not know whether to login, register, book a meeting room, or inspect institution lists first.
- Provincial staff power user: sees a search control and dashboard indicators, but search and progress do not yet feel operationally trustworthy.
- Mobile/field user: must scroll past sidebar and hero before login, then later deals with horizontally scrolling deadline tables.

## Minor Observations

- Bootstrap CSS uses 5.3.7 while footer JS uses 5.3.2. This is usually fine, but aligning versions reduces odd edge cases.
- Sidebar item “แดชบอร์ด” links to `#`, which behaves like a visual placeholder rather than navigation.
- The table overflow is contained, so it does not break the body width, but long mobile tables still feel dense.
- `single-font` is acceptable here because Sarabun is part of the product identity, so I would not treat that as a defect by itself.

## Questions to Consider

1. Should `/main` primarily be a public dashboard, or primarily a login entry screen?
2. Is the institution summary intended to be public before login, or only a preview of operational data?
3. Should the top search become a real menu/data search, or should it be removed until search is implemented?
4. Are the hero progress bars meant to show real percentages, or only decorative activity indicators?
